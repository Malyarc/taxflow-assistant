## HIGH

### H1 — §461(l) TY2026 thresholds are stale: OBBBA reset them to $256,000 / $512,000
`taxReturnEngine.ts:1439-1443` (`SECTION_461L_THRESHOLDS`) holds TY2026 at the TY2025 values ($313k/$626k) with the comment "TY2026 not yet published — held at TY2025." That is no longer true: **OBBBA re-based the §461(l)(3)(B) indexation and Rev. Proc. 2025-32 (published Oct 2025) sets TY2026 at $256,000 single / $512,000 MFJ** — *lower* than 2025, so "hold at 2025" under-states the addback by up to $57k/$114k of income → **under-tax on TY2026 returns** with large business losses. This is exactly the stale-year-map failure class the repo's 2026-06-05c freshness hardening exists to catch — but holding a key *populated* with the wrong value defeats the compile-time guard.
- Fix shape: `2026: { single: 256_000, …, married_filing_jointly: 512_000, qualifying_widow: 512_000 }`.
- Confidence: high (multiple independent sources incl. Tax Notes/Baker Tilly summaries of Rev. Proc. 2025-32).

### H2 — §461(l) auto-aggregation sums per-bucket LOSSES without netting business INCOME (Form 461 requires netting)
`taxReturnEngine.ts:2320-2342`. The auto path computes `aggregateBizLoss = schCLoss + rentalLossPrePal + k1ActiveLoss`, where each term is `max(0, −net)` **per bucket**. §461(l)(3)(A) / Form 461 (lines 1–9) define the excess business loss as aggregate T/B **deductions minus aggregate T/B gross income** minus the threshold — a profitable business offsets a lossy one.
**Live repro (R3):** single TY2024, Sch C profit $400k + active K-1 Box 1 loss $700k → true net business loss $300k < $305k → **Form 461 addback $0**. Engine auto-computes **addback $395,000** (AGI 79,190.50 instead of ≈ −315k) → massive over-tax. Triggers silently (no CPA input) for any multi-entity filer with one large loss and one profitable business — common in the >$305k-loss population this provision targets. The explicit `section_461l_excess_loss_addback` adjustment still wins, but the auto path was specifically built to run when the CPA *didn't* supply it.
Sanity check (R19): pure single-business Sch C loss case is correct (loss $400k → addback $95k, AGI $295k). Sub-finding: Form 4797 §1231 ordinary loss is also excluded from the aggregation (already documented in-code as conservative — but note that with H2 the direction of the documented "conservative" claim was already over-tax, not under).

### H3 — QBI auto-default over-states QBI: SEHI not subtracted; no §199A(c)(2) negative-QBI netting or loss carryforward
`taxReturnEngine.ts:2887-2948, 3025-3055`.
1. **SEHI omission.** Form 8995 instructions / Treas. Reg. §1.199A-3(b)(1)(vi): QBI is reduced by the ½-SE-tax deduction **and the self-employed health-insurance deduction** (and SE retirement contributions) attributable to the business. The auto-default subtracts only ½-SE. **Live repro (R4b):** single TY2024, SE $100k + $150k interest + $20k SEHI → engine QBI deduction **$18,587.05**; correct = 20% × (92,935.23 − 20,000) = **$14,587.05** → $4,000 over-deduction (≈$960–$1,480 under-tax). The engine *computes* `sehi.deduction` two pages earlier — the data is on hand. Masked whenever the 20%-of-taxable cap binds (pure-SE returns), which is why existing tests pass; exposed on any return with other income (W-2 spouse, investment income) — a very common configuration.
2. **No negative-QBI netting (§199A(c)(2)).** An active K-1 Box 1 *loss* flows to AGI but is floored at 0 in the QBI auto-default (`Math.max(0, box1+box3+box2)` per K-1), and a Sch C *loss* (netSeIncome floored at 0) never reduces K-1 QBI. **Live repros:** R5b — SE $100k + active K-1 −$60k → engine deduction $18,587.05, correct $6,587.05 (Form 8995 lines 1–2 net). R18 — Sch C −$40k + active K-1 +$100k → engine $20,000, correct $12,000. Also **no qualified-business-loss carryforward** is tracked anywhere (a net-negative combined QBI must carry forward as negative QBI; engine has no output/CF column for it).
Direction: under-tax (over-deduction). Note: the per-business wage/UBIA `lossScale` machinery (`calculateQbi:6955-6956`) already anticipates netted losses — the netting just never happens upstream in the auto path.

## MEDIUM

### M1 — OBBBA Schedule 1-A: MFS filers wrongly receive tips, overtime, and senior deductions
`taxCalculator.ts:7043-7074`. New IRC §224 (tips) and §225 (overtime) require a **joint return for married taxpayers** (IRS: "must file a joint federal tax return" — MFS ineligible), and the $6,000 senior deduction likewise **cannot be claimed MFS** (categorical, per IRS OBBBA guidance). The engine has no MFS gate: **live repro (R6)** — MFS TY2025 with $10k tips + $5k overtime gets the full $15,000 deduction (statute: $0); an MFS 65-year-old would also get the $6,000 senior deduction. Car-loan interest has no MFS bar (engine's treatment OK). Over-deduction → under-tax for MFS claimants. Also not modeled (LOW, note): the SSN requirement and the tips "qualified occupation" list — CPA-input territory.
Verified-clean in the same function: caps ($25k tips both statuses; $12.5k/$25k OT; $10k car; $6k/senior), phase-out rates ($100/$1,000 over $150k/$300k; $200/$1,000 over $100k/$200k; 6% over $75k/$150k), AGI-based MAGI, taxable-income (not AGI) placement, TY2025–2028 window.

### M2 — §1231(c) lookback recapture must come OUT of the unrecaptured-§1250 (25%) pool first; engine preserves the pool
`form4797.ts:197-207`. The engine recharacterizes the 0/15/20% gain first and keeps `unrecaptured1250Gain = min(pool, survivingLTCG)`, on the stated theory that the IRS worksheet "does not net the §1231(c) recapture against the §1250 amount." **Notice 97-59 and Treas. Reg. §1.453-12 Ex. 3 say the opposite**: net §1231 gain recharacterized as ordinary under §1231(c) is treated as coming first from 28%-rate gain, then from unrecaptured §1250 gain, and only then from adjusted net capital gain. **Live repro (R2):** §1250 sale (gain $40k, all unrecaptured) + land gain $60k + lookback loss $60k → engine keeps **$40k in the 25% bucket**; correct is **$0** at 25% (the $60k ordinary recharacterization absorbs the 25% pool first), with the surviving $40k at 0/15/20. Over-tax up to (25% − 15/20%) × min(pool, recapture). Needs lookback + §1250 + coexisting other §1231 gain to bite — plausible for serial real-estate sellers.

### M3 — §179 carryover is not re-limited by the deduction-year DOLLAR cap
`taxCalculator.ts:5897-5909`. `available = electedAfterDollarCap + carryIn` subjects the carryover only to the income limit. Reg. §1.179-3(b): the carryover deductible in a year is limited to the year's **unused §179 allowance** (dollar limit − current-year elections). **Live repro (R1):** TY2024, $1.22M elected (= cap) + $100k carry-in + ample income → engine deducts **$1,320,000** (law: $1,220,000). Over-deduction; rare at the $2.5M OBBBA caps but a wrong filed number when hit.

### M4 — dollar-cap-disallowed §179 election becomes a §179 carryforward and the excess basis gets NO depreciation
Same function. §179(b)(3)(B) carries forward only the *income-limit* disallowance; an election above the **dollar** limit is simply not expensable — the excess cost recovers through bonus/MACRS in the normal way. **Live repro (R1b):** $1.5M elected TY2024 → engine: §179 $1.22M, **MACRS $0**, "§179 carryforward" $280k. Correct: §179 $1.22M + year-1 MACRS on the $280k (5-yr HY: $56,000). Under-deducts $56k now, over-recovers later through a carryforward channel that doesn't exist in law for this excess.

### M5 — NOL deduction applied at taxable-income level; AGI is never reduced
`taxReturnEngine.ts:2969-2979`. For individuals the §172 NOL deduction is an **above-the-line** item (Schedule 1 line 8a → Form 1040 line 8 → AGI). The engine subtracts it after `calc.taxableIncome`, leaving `adjustedGrossIncome` un-reduced. Headline taxable income is right, but every AGI/MAGI-keyed computation runs on the over-stated AGI: NIIT MAGI, PTC §36B MAGI, medical 7.5% floor, charitable 60/30/50% ceilings (this one *under*-taxes — bigger ceilings), IRA/SLI phase-outs, SALT §164(b)(7) phase-down, OBBBA Sch 1-A phase-outs, EITC AGI test, §469 MAGI. Mostly over-tax. A genuine structural deviation, not just a presentation issue, for any NOL-bearing return with NIIT/PTC/itemized exposure.

### M6 — TY2026 OBBBA charitable provisions not modeled (three items)
`calculateScheduleA` (`taxCalculator.ts:2702-2777`) and the engine have none of the OBBBA charitable changes effective TY2026: (a) the **0.5%-of-AGI floor** on itemized charitable contributions (under-tax for TY2026 itemizer-donors); (b) the **permanent non-itemizer charitable deduction $1,000/$2,000** (over-tax — denies a deduction to every non-itemizing donor; nothing in the adjustment vocabulary models it); (c) the **35%-cap on the benefit of itemized deductions for 37%-bracket taxpayers** (new §68 overhaul; under-tax at the top). All three are TY2026 (planning/projection exposure today; filed-return exposure in 2027). The engine ships "native TY2026," so these are in-scope gaps, same class as the OBBBA items already modeled.

### M7 — Above-the-line §179 adjustment's income limit uses 0.9235×SE earnings and ignores W-2 wages (and K-1 active income)
`taxReturnEngine.ts:2723` — `section179Applied = min(elected, max(0, se.netSeEarnings))`. §179(b)(3)(A) + Reg. §1.179-2(c)(6)(iv): the limit is the aggregate **taxable income from active trades/businesses including W-2 wages** — not net-SE-earnings (the 92.35% Sch SE figure). **Live repro (R8):** $50k Sch C + $100k W-2, $80k elected → engine allows **$46,175** (law: full $80,000 against a $150k limit). Over-tax. Inconsistent with the asset-register path, which correctly adds `totalWages` (`taxReturnEngine.ts:2048-2049`). Also neither path counts K-1 active T/B income in the limit base (conservative).

## LOW

- **L1 — §1202 28%-rate-gain inconsistency.** The adjustment path routes the taxable 50%/75%-exclusion remainder to 0/15/20 (documented sub-gap, repro R7: $500k taxable remainder, 28%-bucket $0 — under-tax ~$40k at the top), even though the 28% bucket now exists (T1.1a); meanwhile the per-transaction `gainClass:"section1202"` path puts the **full** tagged gain at 28% — for 100%-exclusion stock whose only taxable piece is over-cap excess (0/15/20 by law), tagging it over-taxes. The two §1202 channels disagree with each other and each is wrong in a different direction. The in-code comment "a dedicated 28%-rate bucket is the remaining gap" (`taxReturnEngine.ts:2263-2266`) is stale.
- **L2 — Form 8606 input-doc contradiction.** `form8606.ts:48-51` says `totalTraditionalIraBalance` is the Dec-31 balance "(BEFORE the conversion is removed)", but the math (`denominator = yearEndBalance + conversions + distributions`, line 127) and the route (which feeds the H5 year-end asset balance, `routes/tax-returns.ts:453,473`) require the **actual post-conversion Dec-31 balance** (Form 8606 L6; the worksheet adds back L7+L8 itself). Anyone following the interface comment double-counts the conversion in the denominator → under-recovers basis → over-tax. Math itself is correct per L6+L7+L8 given the right input.
- **L3 — Form 8606 Part III is display-only.** `computeForm8606PartIII`'s taxable Roth earnings + 10% penalty are computed at the route; the engine return never includes them (no `roth_ira_distribution` income path in `computeTaxReturnPure`). A CPA could reasonably assume the computed taxable amount feeds the 1040. Ordering rules themselves (contributions → conversions → earnings; §408A(d)(2) qualified shortcut; §72(t) on within-5-yr conversion principal) verified correct.
- **L4 — Carryforward-roster asymmetries.** Auto-load roster (capital-loss ST/LT, PAL, K-1 passive, AMT credit, charitable cash, FTC, adoption, investment interest, R&D GBC, other GBC, SchC §179, NOL, §163(j)) verified present in `synthesizePriorYearCarryforwards`. NOT captured/auto-rolled: **AMT NOL (ATNOLD) remaining** (output field only — not persisted, not seeded); **§1231(c) current-year net loss** (no 5-yr nonrecaptured-loss store; lookback is CPA-supplied); **§461(l) addback → next-year NOL** (documented as CPA-manual, but the auto-aggregation path makes silent under-carry likely); above-the-line `section179Carryforward` (documented "CPA tracks externally"); **negative-QBI carryforward** (see H3); charitable **property** carryforward (documented).
- **L5 — K-1 QBI Box 2 comment/code contradiction + suspended-income QBI.** Comment (`taxReturnEngine.ts:2936-2939`) says Box 2 rental RE is *excluded* from the QBI auto-default; the code includes it (`box1 + box3 + box2`). And because ALL Box 2 routes to the §469 passive bucket for AGI, Box 2 income that ends up suspended still generates QBI — live repro R17: $50k Box 2 fully offset by an $80k suspended passive loss (AGI effect $0) still produced a $10,000 QBI deduction.
- **L6 — Bonus-rate table treats all 2017 acquisitions as 50%** (`BONUS_RATE_BY_ACQUISITION_YEAR`); post-9/27/2017 property was 100%. Affects prior-year basis reconstruction for old assets only.
- **L7 — Two parallel §179 systems** (above-the-line adjustment vs Sch C asset register) each get their own independent dollar cap and income limit — combined elections can exceed the single statutory per-taxpayer §179(b)(1) limit.
- **L8 — QBI cap `netCapitalGain` uses pre-§163(d)(4)(B)-election LTCG/QDIV** (`taxReturnEngine.ts:3064`); the election removes elected amounts from §1(h) net capital gain, so the cap should use post-election figures. Conservative (smaller cap).
- **L9 — Mid-quarter 40% test excludes §280F passenger autos** (`taxCalculator.ts:5783`); Reg. §1.168(d)-1 counts them in the test (they merely keep their own computation capped). Affects only which convention the *other* assets use; documented-adjacent.
- **L10 — Residential/commercial MACRS uses exact fractions** (1/27.5, 1/39 with 12.5-month first-year proration) rather than Pub 946 Table A-6/A-7a rounded percentages — ±$1–2/yr; no mid-month proration in a disposition year (sale-year depreciation runs a full year if not flagged disposed).
- **L11 — §4973(g) HSA excise** ignores the lesser-of-account-FMV cap and multi-year persistence of excess (6% × current-year excess only). Minor.
- **L12 — §461(l) auto-aggregation uses pre-PAL rental losses** (documented in-code; over-addback → over-tax; technically Form 461 takes post-§469 amounts).
